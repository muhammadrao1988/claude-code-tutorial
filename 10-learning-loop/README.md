# Learning Loop (Karpathy-inspired)

## What is it?

The Learning Loop is a pattern where Claude keeps improving something on its own. You set a goal, define how to measure "better," and let Claude run many small experiments. Good changes stick. Bad changes get thrown away.

Andrej Karpathy used this pattern to let an AI agent improve his ML training code overnight. It ran 700 experiments, found 20 real improvements, and gave him an 11% speedup on code he'd already tuned for months.

## Why use it?

Claude can generate many ideas, but it can't easily tell you which ones are actually good. You end up reviewing every suggestion, judging it, and tracking progress. That's slow.

The learning loop fixes this. You define ONE number that measures quality. Claude tries changes, checks the number, keeps the winners. You become the supervisor, not the judge.

## How it works

Five steps, repeated many times:

```
1. Modify   -> Change one thing
2. Run      -> Test it
3. Measure  -> Get a score
4. Decide   -> Better? git commit. Worse? git reset --hard.
5. Repeat   -> Do it again
```

### The five rules that make it work

1. **One metric** -- Pick one number. No arguing about "better."
2. **Small scope** -- Change one section at a time.
3. **Fast verification** -- Keep each test under 5 minutes.
4. **Auto-rollback** -- Bad changes cost nothing because git undoes them.
5. **Git as memory** -- Every commit is a step forward.

## Example 1: For Everyone -- Tune your resume to a job

Drop your resume and a job description into `input/`. Run one command. Get a tuned resume back as a NEW file. Original stays untouched.

### Folder structure

```
examples/non-tech/
├── input/
│   ├── resume.md             # YOUR resume (never modified)
│   └── job-description.md    # the target job
├── output/
│   ├── resume-tuned.md       # the improved version (new file)
│   ├── rubric.md             # auto-generated by an agent
│   ├── score.txt             # current best score
│   └── score-history.md      # what changed each iteration
└── .claude/
    ├── commands/tune.md      # the /tune command
    └── agents/
        ├── rubric-writer.md  # reads JD, writes rubric
        └── tuner.md          # runs the loop
```

### How to use it

```bash
# Copy this example to a working folder
cp -r examples/non-tech ~/my-resume-tuner
cd ~/my-resume-tuner

# Paste your real resume and JD into input/ files
# (edit input/resume.md and input/job-description.md)

# Initialize git (the loop needs it)
git init && git add . && git commit -m "starting point"

# Run the loop
claude
```

Inside Claude Code:

```
/tune
```

Walk away. Come back to `output/resume-tuned.md`.

### What happens

1. The `rubric-writer` agent reads your JD and writes a tailored scoring rubric
2. Claude shows you the rubric and asks if you want to adjust
3. The `tuner` agent runs 15 iterations: rewrite a section -> score -> commit if better, revert if worse
4. You get a summary of what changed and why

For the next job, just swap the JD and run `/tune` again. Same setup.

## Example 2: For Developers -- Tune a skill to pass more tests

Same pattern, different domain. Drop a skill in `input/`. Get a tuned version back that passes more test cases.

### Folder structure

```
examples/dev/
├── input/
│   └── skill.md              # YOUR skill (never modified)
├── output/
│   ├── skill-tuned.md        # the improved version
│   ├── test-cases.md         # auto-generated by an agent
│   ├── score.txt             # current best pass rate
│   └── score-history.md      # what changed each iteration
└── .claude/
    ├── commands/tune.md      # the /tune command
    └── agents/
        ├── test-writer.md    # generates test cases
        └── tuner.md          # runs the loop
```

### How to use it

```bash
cp -r examples/dev ~/my-skill-tuner
cd ~/my-skill-tuner

# Replace input/skill.md with your skill

git init && git add . && git commit -m "starting point"

claude
```

Inside Claude Code:

```
/tune
```

The `test-writer` agent generates test cases from your skill description. The `tuner` agent runs 20 iterations of rewriting the skill to pass more tests. Reported real-world result: a landing-page skill went from 41% pass rate to 92% in 4 rounds.

## Tips

- **The original is sacred.** Both examples never touch `input/`. Tuned versions go to `output/`.
- **One command does everything.** No manual rubric writing or test case design -- agents handle it.
- **Git is the memory.** Each winning iteration is a commit. `git log` shows you exactly what improved.
- **Iterations matter.** 5 iterations barely move the score. 50 iterations can transform it.
- **Adjust the rubric/tests before running.** Both agents pause to let you review and adjust before the loop starts.
- **Token-heavy.** Each iteration is a full Claude run. Use Sonnet, run overnight for big jobs.
